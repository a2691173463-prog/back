# Agent 服务并发改造测试报告

## 1. 测试目的

本测试用于比较 Python Agent 服务改造前后的流式 HTTP 并发能力：

- 改造前：同步 LangChain `model.stream()` + 同步生成器。
- 改造后：异步 LangChain `model.astream()` + FastAPI 异步生成器。

测试关注请求成功率、吞吐量、首 Token 时间（TTFT）和总延迟。它不评价回答质量，也不代表阿里云百炼等真实模型供应商的线上吞吐上限。

## 2. 为什么使用 Mock 模型

真实模型接口会受到网络、限流、排队、账号配额和模型负载影响，直接拿两次线上调用比较，很难判断差异来自代码还是供应商。

因此测试使用项目内的 OpenAI 兼容 Mock 服务，固定：

- 首 Token 延迟：150 ms
- 后续 Token 间隔：40 ms
- 每次输出 Token 数：12
- Uvicorn worker：1
- 请求总数：400
- 并发数：200

这样改造前后面对的是同一套确定性模型延迟。

## 3. 测试工具

- Mock 模型：`agent-service/benchmarks/mock_openai_server.py`
- 改造前兼容服务：`agent-service/benchmarks/sync_stream_server.py`
- 压测客户端：`agent-service/benchmarks/concurrency_benchmark.py`
- 改造后服务：`agent-service/app/main.py`

压测客户端使用 `httpx.AsyncClient` 并发请求 SSE 接口，记录每个请求的首个有效 Token 时间和流结束时间。

## 4. 执行方式

分别启动 Mock、同步基线服务和当前异步服务后，对两个接口各预热一轮，再执行三轮正式测试：

```powershell
uv run python benchmarks/concurrency_benchmark.py `
  --url http://127.0.0.1:18002/llm/chat/stream `
  --requests 400 --concurrency 200 --label before-sync

uv run python benchmarks/concurrency_benchmark.py `
  --url http://127.0.0.1:18001/llm/chat/stream `
  --requests 400 --concurrency 200 --label after-async
```

最终结果采用三轮正式测试的中位数，减少 Windows 本地进程调度造成的偶然波动。

## 7. 结论

本轮测试没有证明单纯改成 `async/await` 能提高吞吐量。两版成功率均为 100%，总延迟 P50/P95 基本持平；异步版吞吐量中位数反而低约 7.8%，TTFT 也有一定波动。

原因是 FastAPI/Starlette 会把同步 `StreamingResponse` 生成器放入 AnyIO 工作线程池，所以旧实现并不是“整个事件循环完全串行”。在当前本机、固定短响应、单进程 Mock 场景中，线程池已经能够并发处理请求，异步调度本身还会产生少量开销。

因此，这次异步改造的已验证收益应表述为：

- Python 模型流、工具 HTTP 和 Checkpointer 形成统一异步调用链。
- 避免每个长连接长期依赖工作线程，为更多慢 I/O 工具和连接打基础。
- 代码可以继续接入异步数据库连接池、超时、取消和背压控制。
- 当前测试验证了 200 并发下 100% 成功，但尚未验证吞吐提升。

不能在简历中写“吞吐量提升 90%”。第一次高并发运行曾出现约 91% 的表面提升，但重复运行后无法复现，属于预热或本地调度波动，未计入正式结果。

## 8. 下一轮如何得到更有意义的数据

下一阶段建议补充资源指标，而不只看 req/s：

1. 用 Windows Performance Monitor、Process Explorer 或 Prometheus 记录 Python 进程线程数、CPU、RSS 内存和事件循环延迟。
2. 把 Mock 单次流持续时间提高到 10 至 30 秒，更接近真实面试长连接。
3. 测试 200、500、1000 个并发连接，观察同步线程池排队和异步连接的资源差异。
4. 增加工具调用场景，模拟简历、画像和题库三个慢 HTTP I/O。
5. 在同一机器、同一进程数下运行至少五轮，并报告中位数和 P95。
6. 最后用小规模真实模型流量验证端到端体验，但把供应商限流与本地服务性能分开解释。

## 9. 面试时建议表述

可以这样讲：

> 我把 Python Agent 的模型流、工具请求和 Checkpointer 改成了异步链路，并搭建了可重复的 OpenAI 兼容 Mock 压测环境。400 请求、200 并发下两版成功率都是 100%。重复测试没有证明吞吐提升，因为 Starlette 会把旧同步流放到线程池中执行。这个结果让我把优化目标从“只改 async”调整为连接资源、线程数、背压和长连接稳定性，避免只凭一次测试宣称性能提升。
